MEPCO Bill Checker — Gaps & Limitations

A deep-dive into the current codebase identifying 11 critical bugs, security issues, and architectural weaknesses.

📅 March 2026 ⏱ 5 min read 🔍 11 Gaps Found
← Back to Bill Checker
🔴 Critical

Gap 1 — Undefined Variable Bug (Server Crash)

In server.js line 160, a variable called referenceNumber is used — but it was never declared. The correct variable is searchValue.

// ❌ Wrong — crashes the server error: `No bill found for reference number ${referenceNumber}.` // ✅ Correct error: `No bill found for reference number ${searchValue}.`

When the MEPCO portal returns a "bill not found" response, this path executes and throws a ReferenceError — returning a 500 error instead of a clean 404.

Fix: Replace referenceNumber with searchValue on line 160.
🟠 High

Gap 2 — New Browser Launched for Every Request

A full Chromium browser is spawned on every single request. This adds 3–5 seconds of overhead, consumes heavy RAM/CPU, and causes server collapse under even light concurrent load.

// This runs on EVERY request 😬 browser = await puppeteer.launch({ headless: true, ... });
Fix: Launch one browser at startup and reuse pages (browser pool pattern).
🟠 High

Gap 3 — No Rate Limiting or Concurrency Control

There is no rate limiter, request queue, or browser cap. Any user can flood the server with hundreds of requests, launching dozens of Chromium instances simultaneously until the server crashes.

Fix: Use express-rate-limit and cap concurrent Puppeteer sessions to 2–3 max.
🟡 Medium

Gap 4 — Fragile Page-Load Detection (Race Condition)

domcontentloaded only waits for raw HTML — not JavaScript-rendered content. The MEPCO portal is ASP.NET AJAX; the bill is injected after the initial HTML. The fallback is a blind 4-second sleep, which is unreliable — sometimes the screenshot fires before the bill is rendered.

// Unreliable fallback .catch(() => new Promise(r => setTimeout(r, 4000)));
🟡 Medium

Gap 5 — Screenshot Only, No Structured Data

The API returns the entire bill as a base64-encoded PNG (often 500KB–2MB). No machine-readable data is extracted — making features like bill history, SMS reminders, or auto-pay impossible to build.

Fix: Parse bill HTML with cheerio and return structured JSON alongside (or instead of) the image.
🟡 Medium

Gap 6 — Retry Button Doesn't Actually Retry

The Retry button triggers checkBtn.click(), which re-validates the current input. If the user has cleared or changed the field, it fails with a validation error instead of repeating the last search.

Fix: Store the last valid { searchType, searchValue } and re-use on retry.
🟡 Medium

Gap 7 — "Download PDF" Actually Downloads a PNG

The button is labeled "Download PDF / Print" but delivers a .png image file. This is misleading and doesn't meet user expectations for an official document.

Fix: Relabel the button "Save as Image" or generate a real PDF using jsPDF.
🟢 Low

Gap 8 — All Navigation Links Are Dead

Every link in the header and navigation bar points to # and goes nowhere. The site presents as an official MEPCO portal but provides no actual navigation — confusing real users.

🟡 Medium

Gap 9 — CORS Is Fully Open

The server uses app.use(cors()) with no origin restrictions — any website on the internet can call this API.

app.use(cors()); // all origins allowed ❌ app.use(cors({ origin: 'https://yourdomain.com' })); // ✅
🟡 Medium

Gap 10 — Browser Memory Leak on Success-Path Errors

browser.close() is only called manually inside the try block. If any exception occurs after a successful fetch but before closure, the Chromium process is left running — accumulating zombie processes over time.

// Move to finally to guarantee cleanup } finally { if (browser) { try { await browser.close(); } catch {} } }
🟠 High

Gap 11 — Vercel Deployment Will Fail

A vercel.json exists in the repo — but Vercel serverless functions have a 50MB size limit while Chromium alone is ~200MB. Deployment will be rejected automatically.

Fix: Deploy on Railway / Render / VPS, or use @sparticuz/chromium for a Vercel-compatible lightweight build.

✅ Conclusion

The core concept is sound and basic functionality works. To make this production-ready, fix in priority order: crash bug (#1) → rate limiting (#3) → browser lifecycle (#2, #10) → deployment issue (#11) → then improve architecture and UX.
